# v11_response_baseline.json source note

`api_contract/v11_response_baseline.json` is the locked compatibility baseline used by `scripts/check_v11_contract_lock.py`.

Source policy:

1. The outer API response shape is the existing v11-compatible backend/frontend contract used by this add-on: `request_id`, `status`, `model`, `input`, `result`, `files`, `runtime`, and `warnings`.
2. The nested `result.raw_summary` structure is populated from the public V11 service visualization summary contract generated by `seg/infer_service_visual.py`.
3. The reviewed public V11 repository snapshot did not expose a captured HTTP `runtime_api/requests/<request_id>/result.json` response sample. If the team has a staging/production V11 response body, replace this file with that exact captured sample and rerun the checker.

The checker compares JSON paths and coarse value types. It does not compare request-specific values such as IDs, file paths, latency, or model version strings.

## Strictness note

By default, `scripts/check_v11_contract_lock.py` compares nested JSON paths and value types against `api_contract/v11_response_baseline.json`, including nested paths under `result.raw_summary`. This matches the conservative "no path/type/nesting drift" policy.

If a future integration intentionally treats `result.raw_summary` as opaque evidence payload, the checker has an explicit escape hatch:

```bash
python scripts/check_v11_contract_lock.py \
  --baseline api_contract/v11_response_baseline.json \
  --candidate runtime_api/requests/<request_id>/result.json \
  --dynamic-object-path '$.result.raw_summary' \
  --require-exact-top-level
```

Do not use that escape hatch when auditing frontend/backend JSON paths that read inside `result.raw_summary`.
