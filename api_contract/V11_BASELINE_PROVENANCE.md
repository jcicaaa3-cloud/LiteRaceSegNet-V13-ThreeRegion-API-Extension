# V11 baseline provenance

This package now separates two different JSON baselines.

## 1. `api_contract/v11_response_baseline.json`

Use this file as the baseline for the legacy-compatible API response body saved at:

```text
runtime_api/requests/<request_id>/result.json
```

This is the compatibility contract that existing frontend/backend consumers should parse. The checker compares JSON paths and JSON value types, not exact runtime values such as `request_id`, file paths, latency, or model version string.

Recommended check:

```bash
python scripts/check_v11_contract_lock.py \
  --baseline api_contract/v11_response_baseline.json \
  --candidate runtime_api/requests/<request_id>/result.json \
  --require-exact-top-level
```

## 2. `api_contract/v11_repo_service_summary_baseline.json`

Use this file only for the service-visualization evidence JSON generated by the legacy V11 service visualization flow. `api_contract/v11_service_summary_reference_from_repo.json` records the extracted reference shape. Typical candidates are:

```text
*_service_summary.json
service_batch_summary.json
```

The public V11 repository exposes this service-summary flow through `run_batch_infer_service.bat`, `seg/capstone_batch_service.py`, and `seg/infer_service_visual.py`. That service-summary JSON is not the same object as the API wrapper response in `result.json`, so do not compare it directly against `runtime_api/requests/<request_id>/result.json`.

Example service-summary check:

```bash
python scripts/check_v11_contract_lock.py \
  --baseline api_contract/v11_repo_service_summary_baseline.json \
  --candidate runtime_api/requests/<request_id>/output/<image_stem>_service_summary.json
```

## Notes

- The V13 model/inference path remains the actual runtime path.
- The V11 repo is used only to identify legacy JSON compatibility surfaces and service-output naming.
- `v13_visualization.json` remains a separate extension artifact and is intentionally not inserted into `result.json` by default.
- The output is a research/capstone/prototype road-damage segmentation visualization artifact, not a certified road safety diagnosis or legal/administrative decision system.

## Checker strictness

The default checker command is strict for the saved outer response and nested paths under `result.raw_summary`. This is the safest mode for the "no path/type/nesting drift" policy:

```bash
python scripts/check_v11_contract_lock.py \
  --baseline api_contract/v11_response_baseline.json \
  --candidate runtime_api/requests/<request_id>/result.json \
  --require-exact-top-level
```

Only use `--dynamic-object-path '$.result.raw_summary'` if the team explicitly documents `result.raw_summary` as an opaque run-specific evidence payload and confirms that legacy frontend/backend code does not parse nested raw-summary fields.
